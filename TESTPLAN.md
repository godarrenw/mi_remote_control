> 对应 DESIGN.md v1.0 · 2026-07-19
> 用法：本清单可直接当验收表用，验收时勾选 checkbox。每条注明类型、前置条件、步骤、预期、边界分支。

**类型图例**

## M1 语音链路（ATVV → ADPCM → BlackHole → 豆包）

### 协议与解码（单元）

- [ ] **M1-01** 〔单元〕 ADPCM 已知向量解码正确性
  - 前置：准备 ≥3 组已知向量（全零输入、正弦波编码样本、open-voice-bridge 抓包实录帧），含期望 PCM 输出。
  - 步骤：喂入 `ADPCMDecoder`，逐样本比对输出。
  - 预期：与参考实现（如 ffmpeg `adpcm_ima_ws`/参考 C 解码器）输出逐样本一致。
  - 边界：每字节高/低 nibble 顺序正确；奇数长度帧；单字节帧。

- [ ] **M1-02** 〔单元〕 ADPCM stepIndex/predictor 钳位边界
  - 前置：构造使 stepIndex 越过 0 和 88 的输入序列；使 predictor 越过 ±32767 的序列。
  - 步骤：解码并检查内部状态与输出。
  - 预期：stepIndex 钳位在 [0,88]，predictor 饱和不回绕（无 wrap 爆音）。
  - 边界：连续最大幅度 nibble（0xF）长串；连续 0x0 长串。

- [ ] **M1-03** 〔单元〕 GET_CAPS 能力帧解析
  - 前置：构造 CTL 0x0B 帧：codec 位含/不含 0x02、不同帧长字段、不同版本号。
  - 步骤：解析并检查产出的会话参数。
  - 预期：codec&0x02 → 16kHz；帧长默认 120B；版本号决定 MIC_OPEN 格式（v≥1.0 用 `0C 00`）。
  - 边界：帧长度不足（截断帧）→ 不崩溃、报协议错误。

- [ ] **M1-04** 〔单元〕 非 16kHz 能力 → 拒绝并重连
  - 前置：CTL 0x0B 帧 codec 不含 0x02。
  - 步骤：注入状态机。
  - 预期：状态机进入断线重连路径，不开启音频会话。

- [ ] **M1-05** 〔单元〕 ATVV 状态机完整时序
  - 前置：Mock peripheral，按 §1.3 时序回放：订阅→GET_CAPS→0x0B→0x08→MIC_OPEN→0x04→音频帧→0x00→MIC_CLOSE。
  - 步骤：驱动状态机走完整会话。
  - 预期：每步写出的 TX 命令字节精确匹配（`0A 01 00 00 03 03`、`0C 00`、`0D <sessionID>`）；sessionID 从 0x04 帧提取并回传。
  - 边界：乱序帧（0x04 之前收到音频帧）→ 丢弃；0x00 后仍到达的音频帧 → 丢弃；未知 CTL opcode → 忽略不崩溃。

- [ ] **M1-06** 〔单元〕 音频帧重组 — 分包
  - 前置：把一个 120B 帧按 BLE MTU 拆成 20B/其他大小的多次 notify。
  - 步骤：依次喂入。
  - 预期：重组出完整帧并解码，样本连续无插入。

- [ ] **M1-07** 〔单元〕 音频帧重组 — 粘包
  - 前置：单次 notify 携带 1.5 帧 / 2 帧数据。
  - 步骤：喂入。
  - 预期：正确切帧，无样本丢失或重复。
  - 边界：帧边界恰好落在 notify 边界。

- [ ] **M1-08** 〔单元〕 音频帧重组 — 丢包
  - 前置：序列中抽掉中间 1 帧（若帧含序号则模拟序号跳变）。
  - 步骤：喂入并继续解码。
  - 预期：不崩溃、不永久失真；收到下一个 0x0A 同步帧后波形恢复正确（与参考对齐）。

- [ ] **M1-09** 〔单元〕 0x0A 同步帧重置解码器
  - 前置：解码器处于任意脏状态；构造带已知 predictor/stepIndex 的 0x0A 帧。
  - 步骤：注入同步帧后继续解码已知向量。
  - 预期：解码器状态被精确重置为帧内值，后续输出与从该状态起解的参考一致。
  - 边界：同步帧字段越界值（stepIndex>88）→ 钳位或拒绝，不崩溃。

- [ ] **M1-10** 〔单元〕 generation 跨代丢弃
  - 前置：模拟连接→断开→重连（generation+1），旧代仍有滞留回调/音频帧。
  - 步骤：投递旧代帧到新代会话。
  - 预期：旧代帧全部丢弃，无串音进入新代音频流。

- [ ] **M1-11** 〔单元〕 [1,2,1]/4 平滑 + 增益
  - 前置：已知 PCM 输入（脉冲、正弦）。
  - 步骤：过平滑与增益链。
  - 预期：脉冲响应为 [1,2,1]/4；增益按配置线性放大；增益后钳位不回绕。
  - 边界：增益=0、增益导致削波、首尾样本（滤波边缘处理）。

- [ ] **M1-12** 〔单元/集成〕 16k→48k 上采样质量
  - 前置：16k 单声道 1kHz 正弦经 AVAudioConverter/AudioBridge 转换路径到 48k。
  - 步骤：对输出做 FFT。
  - 预期：1kHz 主峰完好，镜像/混叠分量 < −40dB；无周期性爆音（buffer 拼接处连续）。
  - 边界：静音输入输出为静音（无 DC 偏置）；连续 30s 转换无 drift 积压。

### 音频设备（集成）

- [ ] **M1-13** 〔集成〕 BlackHole 设备缺失
  - 前置：卸载/未装 BlackHole（或用假设备名测试）。
  - 步骤：启动语音模式 A。
  - 预期：明确报错并在 UI/日志提示"未检测到 BlackHole"，不崩溃；ATVV 不开麦或开麦后音频安全丢弃。

- [ ] **M1-14** 〔集成〕 BlackHole 被占用/采样率被外部改动
  - 前置：另一 app 独占或把 BlackHole 采样率改为 44.1k。
  - 步骤：启动语音链路。
  - 预期：AudioBridge 适配实际采样率或报可读错误；不产生变调（采样率不匹配的典型症状）。

- [ ] **M1-15** 〔集成〕 AudioBridge 启停幂等
  - 前置：无。
  - 步骤：快速 start/stop AudioBridge 20 次。
  - 预期：无泄漏（AVAudioEngine 正常释放）、无残留声音、无崩溃。

### 实机与端到端

- [ ] **M1-16** 〔实机〕 服务发现 UUID 实测确认
  - 前置：遥控器已配对。
  - 步骤：运行服务发现日志工具，列出全部服务/特征 UUID。
  - 预期：存在 `AB5E0001` 服务及 0002/0003/0004 特征（风险表 R1 的固化验证）。

- [ ] **M1-17** 〔实机〕 完整语音会话时序实测
  - 前置：App 运行，日志开启。
  - 步骤：按下语音键说话 3 秒松开。
  - 预期：日志按 §1.3 顺序出现 0x08→MIC_OPEN→0x04→音频帧→（可能的 0x0A）→0x00→MIC_CLOSE；PCM 电平有语音波动。

- [ ] **M1-18** 〔实机〕 蓝牙断连中途恢复
  - 前置：语音流进行中。
  - 步骤：说话途中关闭遥控器电源/走出距离致断连，10 秒后恢复。
  - 预期：App 检测断连、清理会话（generation+1）；回连后再按语音键，新会话正常出声，无旧帧串音。
  - 边界：断连时 AudioBridge 停止写入，BlackHole 无残留循环音。

- [ ] **M1-19** 〔实机〕 语音键快速连按
  - 前置：App 运行。
  - 步骤：1 秒内按放语音键 5 次；再做"松开后立即再按"。
  - 预期：会话正确启停不错乱（MIC_OPEN/CLOSE 成对，sessionID 匹配）；无卡死在开麦状态；最终态与最后一次操作一致。

- [ ] **M1-20** 〔实机〕 长按 >30s 连续采集
  - 前置：App 运行。
  - 步骤：按住语音键持续说话 60 秒。
  - 预期：全程音频连续无累积延迟（首尾延迟差 <200ms）、无内存持续增长、无遥控器侧超时导致的静默中断（若协议有时限，需自动续会话或明确结束提示）。

- [ ] **M1-21** 〔人工〕 豆包端到端出字（M1 验收标准）
  - 前置：豆包输入法麦克风选为 BlackHole 2ch；焦点在文本框。
  - 步骤：按住语音键说"今天天气不错"，松开。
  - 预期：文字正确出现在输入框，延迟 <2 秒；连续 5 次成功 ≥4 次。
  - 边界：轻声/远距离说话仍可识别（增益调节有效）。

## M2 按键引擎（IOHID 独占 + report 解析 + 基础映射）

- [ ] **M2-01** 〔单元〕 report 解析 — usage 表全键
  - 前置：按 §1.4 构造 9 字节 report（reportID=1 + 3 个小端 16-bit usage 槽）覆盖全部 13 键 usage。
  - 步骤：喂入解析器。
  - 预期：每个 usage 映射到正确 ButtonEvent（含按下/松开：槽清零=松开）。
  - 边界：全零 report（全部松开）；未知 usage（如 0x7F 存疑键）→ 上报"未知键"事件供动态学习，不崩溃。

- [ ] **M2-02** 〔单元〕 report 解析 — 多键同按
  - 前置：3 个 usage 槽同时填 ↑ + OK + 音量+。
  - 步骤：喂入，随后逐槽清零。
  - 预期：产生 3 个按下事件；清零顺序对应正确的松开事件；槽位重排（同一键换槽）不产生虚假按放。

- [ ] **M2-03** 〔实机〕 IOHID seize 独占成功
  - 前置：遥控器已连接，输入监控权限已授予。
  - 步骤：启动 App，按方向键/OK。
  - 预期：App 收到原始 report；系统不再响应遥控器按键（如在 TextEdit 中按 OK 不出现回车——除非映射如此配置）。

- [ ] **M2-04** 〔实机〕 双触发检测
  - 前置：seize 成功；映射 OK→回车。
  - 步骤：焦点在文本编辑器，按 OK 10 次。
  - 预期：恰好 10 个回车（系统原生 HID 事件 + App 合成事件不叠加）。
  - 边界：快速连按 20 次仍精确对应。

- [ ] **M2-05** 〔实机〕 seize 失败降级路径
  - 前置：人为使 seize 失败（如撤销部分权限或用调试开关强制降级）。
  - 步骤：按键。
  - 预期：降级为监听 + CGEventTap 吞掉遥控器原生事件，映射仍生效且无双触发；日志明示当前处于降级模式。

- [ ] **M2-06** 〔集成〕 eventSourceUserData 魔数防误吞
  - 前置：降级模式（CGEventTap 激活）。
  - 步骤：App 合成一个与遥控器按键同 keycode 的事件（带魔数）；另用脚本合成不带魔数的同 keycode 事件。
  - 预期：带魔数事件放行（落到前台 app）；真键盘输入不受影响。

- [ ] **M2-07** 〔实机〕 语音键 F5 UserKeyMapping 重映射
  - 前置：App 启动前后对比。
  - 步骤：启动 App，按语音键；观察系统是否触发 F5 默认行为（如听写/键盘快捷键）。
  - 预期：F5 被重映射为无副作用 usage，系统零响应，App 收到语音键事件。

- [ ] **M2-08** 〔实机〕 F5 重映射只影响本遥控器 + 退出恢复
  - 前置：外接真键盘。
  - 步骤：App 运行中按真键盘 F5；退出 App 后分别按真键盘 F5 与遥控器语音键。
  - 预期：真键盘 F5 始终正常；App 退出（含 kill -9 异常退出后重启系统层验证）后遥控器 F5 恢复原始上报。
  - 边界：App 崩溃退出未来得及恢复 → 下次启动时先清理旧映射。

- [ ] **M2-09** 〔集成〕 权限缺失启动行为
  - 前置：撤销输入监控权限。
  - 步骤：启动 App。
  - 预期：不崩溃；`IOHIDCheckAccess` 检出未授权并引导去系统设置；授权后无需重启 App 即可工作（或明确提示需重启）。

- [ ] **M2-10** 〔实机〕 JSON 基础映射生效（M2 验收标准）
  - 前置：config.json 配置方向键→hjkl、OK→回车、返回→Esc。
  - 步骤：焦点在文本编辑器逐键按。
  - 预期：输出与 JSON 完全一致，无双触发。

- [ ] **M2-11** 〔实机〕 未知键动态学习（静音键 0x7F 预案）
  - 前置：进入"按一下识别按键"模式。
  - 步骤：按每个物理键包括存疑键。
  - 预期：UI 显示捕获的 usage 值；存疑键若存在被记录，可绑定动作。

## M3 映射引擎（层/长按/双击/手势/profile）

> 以下状态机测试均可用合成 ButtonEvent + 虚拟时钟驱动，不需要真遥控器。

- [ ] **M3-01** 〔单元〕 tap 基础触发
  - 前置：某键仅配 tap。
  - 步骤：按下→100ms 后松开。
  - 预期：松开时（或若无 double 配置则可即时）触发 tap 动作恰好一次。

- [ ] **M3-02** 〔单元〕 hold 阈值边界 349ms vs 351ms
  - 前置：holdMs=350，键配 tap+hold。
  - 步骤：分别按住 349ms、351ms 后松开。
  - 预期：349ms → tap；351ms → hold（在到达 350ms 时即触发，不等松开）。
  - 边界：恰好 350ms 的行为有定义且文档化（建议 ≥350 判 hold）；hold 触发后松开不再补发 tap。

- [ ] **M3-03** 〔单元〕 hold 阈值可配置
  - 前置：holdMs 改为 200 / 800。
  - 步骤：重复 M3-02 相应边界。
  - 预期：边界随配置移动。

- [ ] **M3-04** 〔单元〕 double 窗口边界（250ms）
  - 前置：doubleMs=250，键配 tap+double。
  - 步骤：两次点击间隔 240ms / 260ms。
  - 预期：240ms → double 恰一次（不发 tap）；260ms → 两次独立 tap。
  - 边界：第一击后等满 250ms 才发延迟 tap（延迟量=doubleMs）；三连击 = double + tap（或有定义的行为）。

- [ ] **M3-05** 〔单元〕 double 关闭时 tap 零延迟
  - 前置：键的 double 配置关闭。
  - 步骤：单击。
  - 预期：松开瞬间立即触发 tap，无 doubleMs 等待。

- [ ] **M3-06** 〔单元〕 momentary_layer 按住进层/松开回落
  - 前置：OK hold=momentary_layer(1)；↑ 配 tap 与 layer:1 两套动作。
  - 步骤：按住 OK 过阈值→按放 ↑→松开 OK→再按放 ↑。
  - 预期：层内 ↑ 触发 layer:1 动作；出层后 ↑ 回到 tap 动作。

- [ ] **M3-07** 〔单元〕 层内按键未松、层键先松开（关键边界）
  - 前置：同 M3-06。
  - 步骤：按住 OK 进层→按下 ↑（不松）→松开 OK→松开 ↑。
  - 预期：↑ 按下时以层内映射结算，松开时仍按**按下时的层**收尾（不得下发层内动作的 down + 基础层动作的 up 之类的撕裂）；无卡键（stuck key）。

- [ ] **M3-08** 〔单元〕 toggle_layer 锁定/解锁
  - 前置：某键 tap=toggle_layer(1)。
  - 步骤：点击进层→按其他键→再点击出层。
  - 预期：层持续锁定至再次点击；层状态变化产生状态回调（供菜单栏/角标）。

- [ ] **M3-09** 〔单元〕 gesture：OK+方向四路
  - 前置：OK 配 gesture 四方向。
  - 步骤：按住 OK→按放 ↑/↓/←/→ 各一次→松开 OK。
  - 预期：四个 gesture 动作各触发一次；松开 OK 不再触发 OK 的 tap/hold。
  - 边界：按住 OK 但不按方向直接松开 → 按 OK 自身 tap/hold 规则结算；gesture 与 hold=momentary_layer 同键共存时优先级有定义且符合文档。

- [ ] **M3-10** 〔单元〕 多键同按映射
  - 前置：↑ 与 OK 各配独立动作。
  - 步骤：↑ 按下→OK 按下→↑ 松开→OK 松开。
  - 预期：两键状态机互不干扰，各自动作正确触发。

- [ ] **M3-11** 〔单元〕 profile overlay 继承
  - 前置：per-app profile 只覆盖 OK 键，`_inherits: global`。
  - 步骤：查询该 profile 下每个键的解析结果。
  - 预期：OK 用覆盖值，其余键与 global 完全一致；多级/循环 `_inherits` 被拒绝或有定义。

- [ ] **M3-12** 〔集成〕 前台 app 切换 → profile 热切换
  - 前置：配置微信 per-app profile。
  - 步骤：脚本激活微信 → 检查生效 profile → 激活 Finder → 再检查。
  - 预期：`NSWorkspace` 通知驱动切换，延迟 <200ms；日志/状态可断言。

- [ ] **M3-13** 〔单元〕 profile 热切换时按键进行中（关键边界）
  - 前置：某键在 global 与 per-app 中映射不同。
  - 步骤：按下该键→注入 profile 切换事件→松开。
  - 预期：本次按键按**按下时的 profile** 完整结算（down/up 配对同源），切换对下一次按键生效；hold 计时不被切换打断或重置出错。

- [ ] **M3-14** 〔集成〕 配置文件热加载
  - 前置：App 运行。
  - 步骤：外部编辑 config.json 改某键映射，保存。
